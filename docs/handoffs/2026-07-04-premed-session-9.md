## What was built
- **`supabase/functions/review-essay/index.ts`** — new Edge Function, the second entry point (alongside the CLI) into the shared rubric/prompt/schema/prose-guard logic in `premed/src/lib/committee-simulator.ts`. Imports that file, `rubrics.ts`, `profiles.ts`, `essay-reviews.ts`, and `school-comparison.ts` **directly via relative path** — no duplicated logic. Exported as a pure `handleReviewEssay(req, deps)` (dependency-injected `supabaseAdmin`/`anthropic`/`now`) with a thin `Deno.serve` wrapper, specifically so it's unit-testable — the two pre-existing functions (`archivist`, `chapter-translator`) are monolithic closures with zero test coverage; this one has 9 handler tests.
  - Auth: identical pattern to the other two functions (`supabaseAdmin.auth.getUser()`, `verify_jwt=true` default).
  - **`profile_id` is always server-derived** from `pm_profiles.user_id = auth.uid()` — never trusted from the request body.
  - Rate limit: counts `pm_essay_reviews` rows for the profile with `created_at` in the last rolling 24h; rejects with 429 at 5. Non-atomic (TOCTOU race exists in theory) — accepted as a low risk since the dashboard button disables while a request is in flight and this is a single-operator surface, not public multi-tenant.
  - Cost gate: reuses the CLI's own `estimateReviewTokens` + `$3/$15 per MTok` + `$5` hard cap, checked before calling Anthropic.
  - Response: `{ essayReview: PmEssayReview, usage }` — the saved row plus token usage, not the bare `EssayReview`.
- **`supabase/functions/review-essay/deno.json`** — import map mapping `zod` → `npm:zod` (the only bare npm specifier the shared lib files actually resolve at runtime — `@anthropic-ai/sdk` and `@supabase/supabase-js` are always `import type`-only in `premed/src/lib/*.ts`, so Deno's TS-stripping erases them and they need no map entry).
- **Three internal `.js`-extension specifiers rewritten to `.ts`** (`committee-simulator.ts`'s import of `rubrics.ts`, and `essay-reviews.ts`/`profiles.ts`'s imports of `schemas.ts`) — the only *value* imports (not type-only) among the shared lib's internal relative imports. `premed/tsconfig.json` already has `moduleResolution: "bundler"` + `allowImportingTsExtensions: true`, and the CLI runs via `tsx`, so this was a zero-risk fix for Node/Vite while making the files Deno-resolvable (Deno does literal-specifier resolution, not TypeScript's `.js`-means-`.ts` NodeNext remapping).
- **`redFlags` added to the shared output schema** (`premed/src/lib/committee-simulator.ts`) — closing session 8's other open item. `redFlags: { key: RedFlagKey, note: string, evidenceQuote: string | null }[]`. `note` is always required (plain-language explanation); `evidenceQuote` is nullable only for `unexplained_gap` (can be inferred from stored activity/profile data the essay never mentions) — `cliche_opening_or_closing` and `professionalism_issue` always require a non-null verbatim quote. Prompt, tool JSON schema, and prose guard (guards `note`, exempts `evidenceQuote`) all extended to match. `report.ts`'s CLI printer and `EssayReviewSection.tsx`'s dashboard rendering both updated.
- **Backward compat for `redFlags`**: the one pre-existing `pm_essay_reviews` row (session 8) has no `redFlags` key in its jsonb. No migration — `listEssayReviews` (`essay-reviews.ts`) normalizes `review.redFlags ?? []` at the read boundary, so every caller (CLI, dashboard) sees a consistent shape.
- **Relocated two CLI-only helpers into shared `premed/src/lib/`** so the CLI and Edge Function call the same code: `aggregateActivities` → `profiles.ts`, `findSchool` → `school-comparison.ts`. `premed/pipeline/review-essay.ts` (the CLI) now imports both rather than defining its own copies.
- **`EssayReviewSection.tsx`** — no longer read-only. New prop contract: `{ reviews, supabase, onReviewSaved: () => Promise<void> | void }`. Textarea + optional school text input + "Get Review" button (disabled while empty/submitting). On submit: `supabase.functions.invoke('review-essay', ...)`, sets `selectedId` to the new review's id *before* awaiting `onReviewSaved()` (a real bug Codex caught in review round 4 — without this, a refetch would prepend the new review but the section would keep showing whichever review was previously selected), surfaces edge-function errors via the same unwrapping pattern as `useChapterTranslator.ts`. Renders `redFlags` (name + note + quote when present) alongside the existing consistency-flags block.
- **`PremedDashboard.tsx`** — wires `EssayReviewSection` with `supabase={supabase}` and `onReviewSaved={real.refetch}` (reuses `useRealProfileData`'s existing `refetch`, the same pattern `ProfileIntakePanel` already uses after a mutation — no new hook surface added).
- **`vite.config.ts`** — added `resolve.alias` mapping the two Deno-only specifiers the Edge Function uses (`npm:@anthropic-ai/sdk@0.96.0`, `https://esm.sh/@supabase/supabase-js@2`) to the already-installed root npm packages, so Vitest can import and test the *actual deployed* `index.ts` file directly rather than a Node-only duplicate.

## The spike that didn't run as planned — and what actually happened instead
The locked plan called for a local go/no-go spike (`deno run` + `supabase functions serve`) before writing the Edge Function. Neither `deno` nor `docker` is installed in this environment, so that spike was impossible to run locally. Flagged to the user mid-session; **chosen path: skip the local spike, deploy directly, and let the real deploy be the test** (a live-and-learn approach, not the original plan's "verify first" — logged as a deliberate deviation, not silently substituted).

- The deploy **succeeded on the first attempt** and correctly bundled the entire dependency graph (`school-comparison.ts`, `rubrics.ts`, `profiles.ts`, `schemas.ts`, `committee-simulator.ts`, `essay-reviews.ts`) — confirming Supabase's function bundler *does* follow relative imports that escape the function's own directory tree. One harmless `WARN` about a stale `.js`-pointing-at-`.ts` type-only reference (the bundler's dependency walker probes even type-only imports, unlike Deno's runtime TS-stripping) — fixed by extending the same `.js`→`.ts` rewrite to the two remaining spots, redeployed clean with zero warnings.
- **Live invocation initially failed** (`"Model call failed"`, no detail surfaced to the browser by design). Diagnosed by reproducing the identical call directly in Node against the real Anthropic key (flagged and approved as a second live spend, ~$0.04) — which **succeeded**, proving `committee-simulator.ts`'s logic itself was correct. Root cause: `archivist`/`chapter-translator` (and the first draft of this function) import `npm:@anthropic-ai/sdk` **unpinned** — Deno resolves npm: specifiers independently of Node's `package-lock.json`, so it can silently drift to a different SDK version than the `^0.96.0` pinned at the repo root. This function is the first one in the repo to use forced tool-calling (`tool_choice`), a surface much more sensitive to SDK version drift than the plain-text-completion calls the other two functions make. **Fixed by pinning the Deno import to `npm:@anthropic-ai/sdk@0.96.0`** (matching the exact version proven to work in the Node diagnostic). Confirmed the fix live (flagged and approved as a third spend, ~$0.04) — full review rendered correctly, including all three red-flag categories and the "500 hours" consistency flag.
- **Recommendation for next session**: consider pinning `archivist`/`chapter-translator`'s `npm:@anthropic-ai/sdk` import too, even though their current plain-text usage hasn't broken — same latent risk, not fixed here (out of scope for this session, same reasoning as leaving their stale `claude-sonnet-4-6` model id alone).

## Non-obvious facts a future session will need
- **The Edge Function's own request-handling order matters and is deliberate**: auth → server-derive `profile_id` → rate limit (cheapest, no external data needed) → resolve activities/school → cost gate (needs the full prompt built) → model call → save → respond. Don't reorder without re-checking the reasoning in `PLAN.md`.
- **`premed/src/lib/*.ts` files must stay import-map-light for Deno**: any *new* bare npm specifier added as a *value* import (not `import type`) to a file the Edge Function transitively imports will need a `deno.json` entry. Type-only imports are free (Deno's TS-stripping erases them before resolution).
- **`vite.config.ts`'s two Deno-specifier aliases exist solely so Vitest can test the real `supabase/functions/review-essay/index.ts` file** — if that file's Deno-only import specifiers ever change (e.g. a version bump), the aliases in `vite.config.ts` must be updated to match exactly, or the Edge Function's tests will fail to transform (hit this once this session — the alias and the specifier drifted out of sync for a few minutes after pinning the SDK version).
- **`typeof Deno !== 'undefined'` guards the `Deno.serve(...)` call** in `index.ts` — required so the file can be imported under Node/Vitest without a `ReferenceError` on the bare `Deno` global.
- **Docker is not installed in this environment**, meaning `supabase functions serve` (local Edge Function testing) is unavailable here — every verification of this function happens via the exported `handleReviewEssay` unit tests plus real deploys. A future session with Docker available could add true local-serve verification as a faster inner loop.
- **`DEV_USER_ID` in `PremedPage.tsx` is untouched** — the Edge Function derives its own user from the real JWT of whichever account is logged into the browser (already how RLS works today), so the browser review button works correctly under the *real* session even though the rest of "real mode" still reads/writes via the hardcoded `devUserId`. Still the most-flagged deferred item across sessions 3-9.
