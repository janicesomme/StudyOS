# Handoff — StudyOS `premed/` subsystem (2026-07-03, end of session 8)

## Scope this session
Committee Simulator — an AMCAS personal-statement critique engine. Read `docs/handoffs/2026-07-03-premed-master-bridge.md` first for cross-session context; this doc only covers what's new.

## What was built
- **`premed/src/lib/rubrics.ts`** — 6 scorable dimensions (theme_coherence, clinical_motivation_shown_not_told, narrative_arc, specificity_evidence, reflection_depth, mission_fit — the last optional, scored only when a target school is given) with 1-5 anchors and adcom-rationale notes grounded in AAMC's Core Competencies framework and AMCAS W&A "show don't tell" guidance. Plus 3 red-flag categories (cliche, unexplained gap, professionalism issue) — currently reference data only, not yet wired into the model's output schema.
- **`premed/src/lib/committee-simulator.ts`** — prompt builder, Zod schema for the structured output, the critique-only hard rule (stated in the prompt + enforced structurally: no rewrite field exists), and a >15-word prose-strip guard (`applyProseGuard`/`truncateToWords`) applied to every generated-commentary field. `evidenceQuotes` are exempt from the guard — they must stay verbatim, they aren't generated prose. Schema refines that a weak dimension (score ≤3) must carry a non-null `challengeQuestion`.
- **`premed/src/lib/essay-reviews.ts`** — persistence (`saveEssayReview`/`listEssayReviews`) against the new `pm_essay_reviews` table, plus `hashEssay`.
- **`supabase/migrations/20260703020000_premed_essay_reviews.sql`** — `pm_essay_reviews` (id, profile_id, essay_sha256, rubric_version, scores jsonb, review jsonb, model, created_at), user-owned via the same `pm_profiles` join-based RLS tier as `pm_activities`/`pm_narratives`. **Applied by the user in the Supabase SQL editor this session** (same manual-apply pattern as every prior migration in this project).
- **`premed/pipeline/review-essay.ts`** (`npm run review-essay -- --user <id> --file draft.txt [--school "..."] [--go]`) — cost-gated CLI (same `--go` protocol as `scrape-class-profiles.ts`), resolves the profile + activities for consistency checking, does a fuzzy `ilike` school lookup for mission_fit, one Sonnet call, saves + prints the review.
- **`premed/src/ui/sections/EssayReviewSection.tsx`** — dashboard section wired into `PremedDashboard.tsx`, real mode only. Renders review history (clickable list) with dimension score bars, quoted evidence, challenge questions, strengths/fixes, consistency flags, verdict.
- Test fixture: `premed/pipeline/__fixtures__/flawed-personal-statement.txt` (734 words — cliché opening/closing, told-not-shown motivation, an unsupported "500 hours" claim for the consistency check to catch).

## Deliberate scope deviation — dashboard has no live "review" button
The original scope asked for a paste-in textarea + review button on the dashboard. **This was not built.** Every other paid-API call in this repo (`scrape-class-profiles.ts`) runs server-side via a Node CLI with the key in `.env.local` — there is no Express/API-route backend or Supabase Edge Function anywhere in this project. A browser button calling Anthropic directly would require shipping `ANTHROPIC_API_KEY` in the client bundle: a real vulnerability (anyone could extract it via devtools and run up spend on the account), not a style nitpick. Flagged to the user mid-session; the user chose **CLI-only generation, dashboard is read-only** for essay reviews (option recommended and selected). If a future session wants an in-browser review button, it needs one of: a minimal Express/Node proxy, or a Supabase Edge Function — both are new deployment surfaces this repo doesn't have yet, and that decision should go through the same kind of check-in.

## Non-obvious facts a future session will need
- **`premed/src/lib/*.ts` must stay browser-safe (no Node built-ins)** — this was nearly broken this session. The first draft of `essay-reviews.ts` used Node's `crypto.createHash`, which Vite silently externalizes for the browser build (no error, just a build-time warning) rather than failing hard — `npm run build`'s output is the only place this surfaced. Fixed by switching `hashEssay` to the Web Crypto API (`globalThis.crypto.subtle.digest`), which works identically in Node and the browser and made `hashEssay` async. **Lesson: after touching any `premed/src/lib/*.ts` file, check `npm run build` output for "externalized for browser compatibility" warnings, not just typecheck/tests** — typecheck and vitest (both Node-context) don't catch this class of bug.
- **Model ID**: the session scope said `claude-sonnet-4-6`; that isn't a real model ID. Used `claude-sonnet-5` instead, matching both this repo's existing `SONNET` constant (`scrape-class-profiles.ts`) and the current model roster.
- **`review-essay.ts`'s school lookup is fuzzy** (`ilike '%query%'`, first match wins) since `pm_schools.name` has no unique constraint and school names are long ("Tulane University School of Medicine"). Fine for a single-operator CLI; would need tightening (exact id, or disambiguation on multiple matches) before any multi-user surface used it.
- **`pm_essay_reviews` is append-only** — every review is a new row, never an update. Review history across drafts is the point (has this dimension improved?).
- Red flags (`RED_FLAGS`/`RED_FLAG_KEYS` in `rubrics.ts`) are defined but **not yet wired into the model's output schema or the UI** — the model currently surfaces cliché/gap/professionalism issues only indirectly through `priorityFixes`/`verdict`. A future session could add a dedicated `redFlags: RedFlagKey[]` field to `EssayReviewSchema` if that granularity turns out to matter.
