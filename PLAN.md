# Plan: /premed goes multi-user (session 10)
_Locked via grill — by Claude + Janice_

## Goal
Wire `/premed` to StudyOS's real, already-built auth system (`useAuth`, `LoginForm`/`SignupForm`, session persistence — all of this already exists at the app level) instead of the hardcoded `DEV_USER_ID`. Make `/premed` itself the front door: logged-out visitors see a one-line pitch, the full demo-mode experience (archetype switcher + read-out), and an inline signup/login form, all on the same page, without ever leaving it. Logged-in visitors see the exact dashboard session 9 built, now correctly scoped to their own `auth.uid()`. No schema changes — RLS has been `auth.uid()`-scoped since session 1; this session is wiring, not data-model work.

## Approach

1. **Remove `/premed` from `App.tsx`'s app-wide `ProtectedRoute`** — it becomes a public route. `PremedPage.tsx` itself takes over gating via `useAuth()`.

2. **`PremedPage.tsx` rewritten** to branch on `useAuth()`'s `{ session, loading, signIn, signUp, signOut }`:
   - `loading`: existing app-wide loading treatment (same spinner/text pattern `ProtectedRoute` already uses).
   - `session` present: render `DashboardShell` + `PremedDashboard`, passing `userId={session.user.id}` (replacing `DEV_USER_ID` entirely — delete the constant and its comment).
   - `session` absent: render `PremedDashboard` directly (no `DashboardShell` — that component's nav/sign-out doesn't apply to a logged-out visitor) with `userId={null}`, plus a one-line pitch above it and a `loggedOutSlot` prop containing the inline `LoginForm`/`SignupForm` (imported from `src/components/auth/`). **Not a raw prop pass** — `SignupForm` emits `(name, email, password)` while `useAuth().signUp` expects `(email, password, name)`, a parameter-order mismatch an adversarial pass caught. `PremedPage` supplies explicit adapter handlers: `handleLogin(email, password) => signIn(email, password)`, `handleSignup(name, email, password) => signUp(email, password, name)`. No navigation call in either — the session updating is what flips `PremedPage` into the logged-in branch automatically.
   - **Email confirmation**: `useAuth.signUp` is widened to return `{ error: Error | null, session: Session | null, user: User | null }` (it already receives `data.session`/`data.user` from `supabase.auth.signUp()` today, they just weren't being surfaced — a gap an adversarial pass caught, since `PremedPage` can't branch on a session that was never returned, and the live RLS check in step 6 needs the new user's id even when confirmation blocks a session). `handleSignup` checks the result: if `session` comes back, nothing further needed (the branch flips automatically); if not, show a "check your email to confirm your account" message in place of the form instead of silently doing nothing.
   - **`SignupPage` updated in the same pass** (an adversarial pass caught that declaring it "unaffected" was wrong: if confirmation is enabled, it would still navigate to `/dashboard` on a no-session success today, causing a confusing bounce back to `/login` via `ProtectedRoute`) — it now branches on `session` exactly like `PremedPage`'s `handleSignup`: navigate to `/dashboard` only when a session comes back, otherwise show the same "check your email" message. Same fix pattern applied twice, not new design, and cheap since this session is already changing `useAuth.signUp`'s return shape.
   - **Sign-out from `/premed`**: `DashboardShell` gains a new optional `onSignOut?: () => Promise<void> | void` prop (backward-compatible — every other page keeps today's exact behavior when it's omitted; typed as async-capable and awaited before any `navigate()` call, since `auth.signOut()` is itself async — an adversarial pass caught that a synchronous-only type would risk not awaiting it). When provided, its sign-out button calls that instead of `signOut()` + `navigate('/login')`. `PremedPage` passes `onSignOut={() => auth.signOut()}` (no navigation) — resolves a contradiction an adversarial pass caught between "keep using `DashboardShell`" and "stay on `/premed` after sign-out," since `DashboardShell`'s hardcoded redirect would otherwise override that.

3. **`PremedDashboard.tsx` prop contract changes**:
   - `devUserId: string` → `userId: string | null`. `null` means logged out.
   - New optional `loggedOutSlot?: ReactNode` — when `userId` is `null`, the "Real profile" toggle button is hidden (only "Demo" shows).
   - **Rendering derives `effectiveMode = userId ? mode : 'demo'`** (not raw `mode` state) for the demo-vs-real content switch — closes a bug an adversarial pass caught: without this, stale `mode: 'real'` state from before a sign-out would survive the toggle being hidden and drive broken rendering against a null `userId`.
   - **`loggedOutSlot` renders independently of `effectiveMode`, not nested inside a `mode === 'real'` branch** — a second, more serious bug an adversarial pass caught: the plan's first draft implied `loggedOutSlot` would render "where the real-mode content would otherwise go," but forcing `effectiveMode` to `'demo'` for every logged-out visitor means that branch never executes, so `loggedOutSlot` would never appear at all. Corrected: when `userId === null`, the demo archetype switcher + read-out renders (via `effectiveMode === 'demo'`) *and* `loggedOutSlot` renders in its own dedicated slot alongside it — both visible together, one is not conditional on the other.
   - **`PremedDashboard` does not import `useAuth` or anything from `src/hooks`/`src/components`** — `loggedOutSlot` is a plain `ReactNode` prop. This preserves the existing clean module boundary between `premed/src/ui/` and the main app's `src/` (established explicitly in session 6's dashboard-integration commit) — auth wiring stays owned by `PremedPage.tsx` in `src/pages/`.
   - **`useRealProfileData` widened to accept `userId: string | null`, not conditionally called** — React hooks can't be called conditionally, so the hook itself must handle `null` internally. The `userId === null` branch returns exactly: `{ profile: null, activities: [], activityGaps: null, poolPositions: null, gapAnalysis: null, essayReviews: [], loading: false, error: null, refetch }` — no Supabase calls, and `loading` explicitly `false` (the hook's current initial state defaults `loading: true`, so an unspecified null branch was flagged as a real risk of a permanently-loading logged-out UI).
   - **Stale-state race guard, specified concretely**: a `currentUserIdRef` (or equivalent) is updated at the start of every `refetch`/effect run and checked immediately before every `setState` call inside that fetch; any commit where the ref no longer matches the fetch's own captured `userId` is dropped. Closes a race an adversarial pass caught, where a slow in-flight fetch for user A could otherwise commit state after sign-out or after switching to user B.

4. **First-run behavior**: no new UI. `PremedDashboard`'s existing `hasReadOut` gate (hides percentile/gaps/school-comparison/essay-review sections until `gpa_cum`+`mcat_total` are set, while `ProfileIntakePanel` always renders) already satisfies "new user lands on the intake form, not an empty dashboard." Verified live with a brand-new second account (step 6) rather than rebuilt.

5. **Rate limit**: no code change — the Edge Function already derives `profile_id` from the JWT (session 9). Verified by re-running its existing unit tests (already cover the ≥5-in-24h rejection with mocked deps) and confirmed incidentally during the live RLS check (step 6) that two different real accounts get independently-scoped rate limits (different `profile_id`s). Not stress-tested to the actual cap live — that would mean 5 real paid calls for no new information the unit tests don't already provide, against the "zero API cost except one approved live check" rule.

6. **Live two-account RLS verification**:
   - Create a brand-new second real account through the actual `/premed` signup form built in step 2 (e.g. `premed-rls-check@test.local`) — exercises the real feature end-to-end rather than repairing session 8's seeded `archetype-grinder@premed.test` (which exists in `auth.users` but has an unknown password) for an unrelated purpose.
   - **Email-confirmation contingency** (an adversarial pass flagged that a `test.local` address can never confirm via a real inbox if the project requires it): attempt the real signup first. If the response comes back with no session (confirmation required), immediately call `supabase.auth.admin.updateUserById(id, { email_confirm: true })` via the service-role key on that *same* just-created account, rather than abandoning the real-signup path for a fully admin-created one — still exercises the actual signup form (the point of this decision), while unblocking the login needed for the RLS check itself. Decided empirically from what the signup call actually returns, not guessed in advance.
   - **Corrected methodology** (an adversarial pass caught that the original wording — "direct anon-key query with no JWT" — only proves anonymous access is blocked, not that account A specifically can't see account B): with two real logged-in sessions, query **as account A's authenticated session** for account B's *specific known* `pm_profiles`/`pm_activities`/`pm_essay_reviews` row ids, expecting empty/blocked results — not an unauthenticated query.
   - **Write checks, not just reads** (an adversarial pass caught the original plan only covered cross-account reads, while the dashboard also writes to all three tables): as account A, also attempt an update/insert/delete against account B's known row ids on the same three tables, expecting RLS to block every one — not the service-role bypass every CLI in this repo has used so far.
   - **One approved live essay-review call** may be used here if needed to populate a second account's `pm_essay_reviews` row to confirm cross-account isolation on that table specifically (the only paid-API surface touched this session) — otherwise this check is API-cost-free. Record the profile ids and request status involved in the handoff (no essay text) for a durable record of what was actually checked.

7. **Tests**: `PremedPage`-level tests for the three render branches (loading / logged-out landing / logged-in dashboard), `PremedDashboard` tests for the `userId=null` branch (toggle hidden, `loggedOutSlot` rendered, `effectiveMode` forced to `'demo'`) and `userId` present branch (existing behavior unchanged, just prop-renamed). A new `App.tsx` routing test confirming `/premed` renders without redirecting when logged out, while `/dashboard` (and other `ProtectedRoute`-wrapped routes) still redirect to `/login` — closing a gap an adversarial pass flagged (no test previously locked in that only `/premed` becomes public). Before claiming "all existing tests stay green": grep every `devUserId` usage across the repo (test files included) and run a full typecheck as an explicit verification gate, rather than asserting it optimistically off the prop rename alone.

8. **Handoff**: `docs/handoffs/2026-07-04-premed-session-10.md`.

## Key decisions & tradeoffs
- **`/premed` leaves the app-wide `ProtectedRoute`** and becomes self-gating — the only way to satisfy "demo mode fully usable while logged out." Every other route's gating is untouched.
- **Inline auth, not a redirect** — `LoginForm`/`SignupForm` are reused directly inside `PremedPage`'s logged-out branch, wired straight to `useAuth().signIn`/`signUp` with no `navigate()` call. The existing full-page `/login`/`/signup` routes (which redirect to `/dashboard` on success) are untouched and still serve the rest of the app; `/premed` just doesn't use them for its own inline form.
- **Sign-out from `/premed` stays on `/premed`** — deliberately different from `DashboardShell`'s default (redirect to `/login`), because bouncing a signed-out visitor away from the page that's supposed to be the "hook" defeats the point.
- **`PremedDashboard` reused for the logged-out demo view, not duplicated** — gains a `userId: string | null` + `loggedOutSlot?: ReactNode` prop contract rather than a second, separate demo-rendering component. One implementation of the archetype switcher and read-out sections. Costs a small conditional (hide the "Real profile" button when `userId` is null, derive `effectiveMode` instead of trusting raw `mode` state) inside an already-existing component, in exchange for zero risk of the two demo views drifting apart over time.
- **`useRealProfileData` accepts `userId: string | null` and no-ops internally rather than being conditionally called** — React's rules of hooks don't allow conditional invocation; the null-handling has to live inside the hook. Also gained a same-userId-at-commit-time guard on its `refetch`, closing a real (if narrow) stale-state race between sign-out/account-switch and an in-flight fetch.
- **Module boundary preserved**: `PremedDashboard`/`premed/src/ui/*` never import `useAuth` or anything from `src/`. All auth wiring lives in `PremedPage.tsx` (`src/pages/`), passed down as plain props/`ReactNode`. Matches the boundary session 6 explicitly established.
- **Second RLS test account created via real signup**, not by resetting the session-8 seeded account's password — tests the actual new feature, doesn't mutate an existing test fixture for an unrelated purpose.
- **Rate limit is verified, not re-tested against its actual cap live** — unit tests already cover the ≥5 rejection; spending 5 real Anthropic calls to re-confirm that live would violate the session's "zero API cost except one approved check" rule for no new information.
- **No distinct "first-run" UI** — the existing profile-null / `hasReadOut`-gated rendering already does the job; this session verifies it live with a fresh account rather than building something new.

## Risks / open questions
- Exact copy for the one-line pitch and the logged-out header/nav treatment (since `DashboardShell` isn't used there) is left to implementation judgment — low-stakes, reversible UI text/layout, not a decision this plan needs to pre-commit.
- Whether `PremedDashboard`'s internal layout needs restructuring beyond the toggle-hide + slot-swap to look intentional (not just "the real-mode UI with a hole in it") for logged-out visitors is a judgment call at implementation time, reversible without re-architecture.
- Whether this Supabase project requires email confirmation on signup is unconfirmed (not in `supabase/config.toml` — likely a dashboard-level setting). Handled defensively (branch on whether a session comes back) rather than assumed either way.
- `EssayReviewSection` mid-submit sign-out (a request in flight when the session clears) is an accepted, unaddressed edge case — extremely low probability for a single-operator testing session, and React 18 silently no-ops a state update on an unmounted component rather than erroring. Flagged, not engineered around.
